## Compiler Review (Round 2)

### What's Fixed Since Last Review
The three highest-priority items from round 1 are addressed:
- **#1 `days_per_unit` vs `days_per` bug** — fixed with a single inline `days` function using 365.2425 everywhere
- **#2 CSV reader extraction** — `read_csv_rows` callback pattern replaces the triplicated file I/O
- **#4 `expanded_comp_cache` made non-optional** + full Hashtbl-based O(1) lookups (`let_tbl`, `comp_tbl`, etc.) via `build_lookup_tables`
- **#3 serde co-location** — serialize/deserialize combined into `serde.ml` with clear contract documentation
- **Dimcheck implemented** — 662 lines with 805 lines of tests, covering the full design doc

### New Bugs

**1. Rust IR missing `param_dim` field (wire format mismatch).** The OCaml side serializes `param_dim` to JSON, but `parameter.rs` doesn't have the field. Serde's default behavior silently ignores unknown fields, so this doesn't crash — but it means `param_dim` is lost in the OCaml→Rust pipeline. The Rust side should add `pub param_dim: Option<(i32, i32)>` with `#[serde(default)]` for completeness.

**2. Dimcheck `Const/Const` special case may over-infer.** The `is_bare_const` check treats `Const(x) / Const(y)` as a fresh variable, assuming it's an inline rate like `1/730`. But this also matches things like `Const(100000)` used as a denominator in `2.3 'per_year / 100000`. That expression would give the division a fresh variable rather than `T⁻¹ / dimensionless = T⁻¹`. The propagation may resolve it correctly, but it's a fragile heuristic — it depends on the caller providing enough context to pin the variable.

**3. Dimcheck `Unknown` linking is incomplete.** When `unify` encounters `Unknown id1, Unknown id2`, it returns `Unknown id1` without linking the two. This means if both are later resolved independently, no consistency check happens between them. For example, if variables `?1` and `?2` are unified, then `?1` is resolved to `T⁻¹` and `?2` to `P`, there's no error. The fix is a union-find structure, or at minimum linking `id2 → id1` in the resolved table.

### UX Issues for Modelers

**4. The `loss` transition in `sir_dim_annotated.camdl` has a dimensionally unusual rate.** `loss : S --> @ coupling * beta * S` — this is `dimensionless × T⁻¹ × P = P·T⁻¹`, which checks out. But from a modeler's perspective, `coupling * beta` is a modified rate, and it's not obvious whether `coupling` should be dimensionless or have its own dimension. The dim checker correctly infers it, but there's no feedback to the user about *what* was inferred for `coupling`. Running `camdl inspect --params` should show inferred dimensions.

**5. `k : real` in the observations model gets no dimension check.** The NegBin dispersion parameter `k` is `kind = real` with no annotation. The dimcheck correctly constrains it to be dimensionless (via the `NegBinomial.dispersion` check), but the user gets no feedback that this happened. If they accidentally made `k : count`, it would still pass the obs model check (since `count` = P, and the check only fires on non-dimensionless). Actually wait — looking at the code, the check is `dim_is_zero v` which IS the dimensionless check. So `k : count` WOULD trigger E307. Good.

**6. Parser `dim_literal` uses `failwith` instead of parser errors.** The `dim_literal` rule calls `failwith` for unknown dimension names (e.g., `[Q]`). This produces an uncaught OCaml exception rather than a clean diagnostic with source location. Should use the Menhir error recovery mechanism or at minimum catch the exception in `compile`.

**7. No `[P^2]` or higher-order dimensions in parser.** The `dim_literal` grammar handles `P`, `T`, `P*T^-1`, `P/T`, `1/T`, `T^-1` — but not `P^2`, `T^2`, or `P^2*T^-1`. These are rare in epi but would be needed for variance parameters (dimension P²) or diffusion coefficients. Low priority but worth noting as a parser limitation.

**8. `simplex_member` param_kind maps to dimensionless.** This is correct (simplex members are fractions summing to 1), but if a modeler uses `simplex_member` for age-specific population fractions that are later multiplied by N to get counts, the expression `frac * N` has dimension `1 × P = P`. This is fine for a rate like `mu * frac * N = T⁻¹ × 1 × P = P·T⁻¹`, but the modeler might expect `frac` to carry population dimension. The current design is correct; just documenting for completeness.

### Code Quality

**9. Dimcheck re-infers on every check.** In pass 2 (constraint checking), `check_model` calls `infer st ~ctx tr.rate` again for every transition — but `infer` has side effects (it emits constraints and creates fresh variables). This means pass 2 adds MORE constraints and variables to the state, potentially causing different results than pass 1. The check phase should read resolved dimensions without re-inferring. Fix: cache the dimension of each transition's rate after pass 1, then check the cached value in pass 2.

**10. Test golden infrastructure is solid.** The golden file test harness with skip-on-missing-data and the negative golden test pattern are well designed. The QCheck property tests are a nice touch, though the `zero_is_universal_additive_identity` property tests arithmetic on dim_vecs rather than the actual `Any` variant behavior in the checker.

### DSL UX Issues for Modelers

**11. Init block uses mangled names instead of indexed syntax.** In `polio_spatial_5.camdl`:

```
init {
  S[north]  = N0_north  - I0
  S[south]  = N0_south
```

Compartments use `S[north]` (indexed), but parameters use `N0_north` (mangled). This is inconsistent — the user declares `N0[patch]` in the parameters block but must refer to the expanded name in init. The indexed syntax `N0[north]` should work here too. This is the single biggest UX papercut in the DSL: the user thinks in terms of `N0[patch]` everywhere but must mentally mangle to `N0_north` in some contexts.

**12. Seasonal forcing dimension ambiguity.** In `seir_seasonal_patch.camdl`:

```
forcing {
  seasonal[p in patch] : sinusoidal {
    amplitude = amp[p]
    period    = 365.0
    baseline  = baseline
    phase     = 0.0
  }
}
transitions {
  infection[p in patch] : S[p] --> E[p]
    @ seasonal[p] * S[p] * I[p] / N[p]
}
```

Here `baseline : rate` and `amplitude : positive`, so the sinusoidal output has dimension T⁻¹. The rate expression `seasonal[p] * S[p] * I[p] / N[p]` = T⁻¹ · P · P / P = P·T⁻¹ ✓. This is correct, but it's non-obvious: the modeler might expect the sinusoidal to produce a dimensionless multiplier (0.6 to 1.4) that scales a separate rate parameter. A modeler coming from pomp would write `beta * (1 + amplitude * sin(...)) * S * I / N`. The camdl version bakes `beta` into `baseline`, which is valid but unfamiliar. A note in the language spec about this pattern would help.

**13. `scenarios` set block uses mangled param names.** In every golden model:

```
scenarios {
  baseline {
    set = {
      amp_urban  = 0.4
      amp_rural  = 0.2
      N0_urban   = 100000
```

Same issue as #11 but in a different context. Users declare `amp[patch]` but write `amp_urban` in scenarios. This should support `amp[urban] = 0.4`.

**14. No way to express `sum` over a subset of dimensions.** The `sum(b in age, ...)` construct sums over ALL levels of a dimension. If a modeler wants to sum over a subset (e.g., "infectious stages only"), they must enumerate manually. This is a minor limitation but comes up in Erlang-staged models where you want total infectious = `sum` over I substages but not E substages.

**15. `where` guard error messages lack source location.** When a `where p != q` guard filters a transition, the user gets no feedback about HOW MANY transitions were filtered or which combinations were excluded. The `--summary` output shows the filter count, but during compilation there's no indication that the guard is doing what the user intended. A model with a typo in a guard name silently produces zero transitions.

### Dimcheck Implementation Issues (Continued)

**16. E303 (cross-transition parameter inconsistency) is never emitted.** The error message catalog in the design doc specified E303 for the case where a parameter is inferred as T⁻¹ in one transition and P in another. The implementation relies on the global constraint system to surface this as E302 (add/unify mismatch) or E300 (wrong rate dimension). The test explicitly documents this:

```ocaml
(* E303 manifests as E302 in the current checker *)
```

This is a diagnostic quality issue. When E302 fires because of a cross-transition conflict, the user sees "dimension mismatch in addition" pointing at ONE expression in ONE transition, with no hint that the root cause is inconsistent usage across transitions. Adding an explicit E303 check (compare inferred dimensions per-parameter across transitions after solving) would dramatically improve the error experience.

**17. Dimcheck `Unknown/Unknown` unification drops constraints.** When `unify` encounters two unknown variables, it returns `Unknown id1` without linking them:

```ocaml
| Unknown id1, Unknown id2 ->
    if id1 = id2 then Unknown id1
    else Unknown id1  (* can't link yet *)
```

This means if `?1` and `?2` are unified (must be equal), then `?1` is later resolved to T⁻¹ and `?2` to P, there's no error — the linking was lost. The fix is either union-find (canonical approach) or at minimum recording `(id2, id1)` in a separate linking table and checking during resolution. For most models this doesn't manifest because the 3-round iteration + propagation resolves variables from known ground truth before two unknowns meet. But it's a correctness gap that could produce false negatives on complex models with chains of unknown parameters.

**18. Dimcheck re-infers during the check phase.** Pass 2 calls `infer st ~ctx tr.rate` again for every transition. Since `infer` has side effects (creates fresh variables, emits E302 for mismatches), this adds spurious constraints and potentially duplicate error messages. The fix: after the 3-round inference loop, read the resolved dimension of each rate from the state without re-inferring. Add a `read_dim st expr` function that only resolves (no fresh vars, no side effects).
