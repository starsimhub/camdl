## Compiler Review

Reviewed: lexer.mll, parser.mly, expander.ml, autodiff.ml, dimcheck.ml,
serde.ml, compiler.ml, and the Rust ir crate.

### Architecture

The pipeline (Lexer â†’ Parser â†’ AST â†’ Expander â†’ IR â†’ Autodiff â†’ Dimcheck â†’
Serialize) is well-separated. The 2077-line expander is the right size for
what it does. The hand-written serde.ml (949 lines) maintains exact control
over the cross-language JSON wire format and matches the Rust IR crate's
untagged serde pattern correctly â€” traced through every major type's
serialization on both sides. The round-trip test (8 golden models) validates
this.

### What's done well

- Dimensional analysis (dimcheck.ml, 903 lines, 70 tests) with two-pass
  inference and proper error messages
- Symbolic autodiff with algebraic simplifier applied to fixed point
- Cross-language IR contract with hand-written serde ensuring OCaml and
  Rust agree exactly
- Error quality where the Diagnostics module is used â€” source locations,
  underlines, hints, related locations
- 118 test cases across compiler and dimcheck, covering golden models from
  trivial SIR to 5-patch spatial SEIR with interventions
- Guard evaluation correctly filters transition expansions at compile time
- Table lookup linearization correctly computes row-major indices for
  multi-dimensional tables
- normalize_expr optimization collapsing Pop + Pop + ... into PopSum

### Issues

**#1 (Critical): Indexed observations not expanded.**
`expand_observations` (expander.ml:1788) uses `List.map` â€” one output
observation per input declaration. It never calls `cartesian_product` on
`od.oindices`. Compare with `expand_transitions_counted` (calls
`cartesian_product tr.trindices`) and `expand_scheduled_actions` (calls
`cartesian_product iv.ivindices`). A model with:

```
observations {
  cases_by_patch[p in patch] : {
    projected = incidence(infection[patch = p])
    ...
  }
}
```

Gets one observation named `cases_by_patch` with unresolved `p`, not N
observations per patch. The spec (Â§13.3) documents "generates one
observation stream per patch." No golden test or unit test covers indexed
observations. This blocks spatial inference models from compiling correctly.

Fix: Replace `List.map` with `List.concat_map` using the same
`cartesian_product` + `filter_map` + guard pattern used in
`expand_transitions_counted` and `expand_scheduled_actions`. Pass the env
through to `resolve_expr`, `index_item_to_str`, and all projection/likelihood
resolution. Generate one `Ir.observation_model` per combination with mangled
names (`cases_by_patch_north`, `cases_by_patch_south`, etc.).

**#2 (High): `resolve_float_expr` silently returns 0.0.**
Lines 1225-1230: any non-Const IR expression returns 0.0. Affects: `simulate
{ to = ... }`, observation schedules, intervention schedules, time function
arguments. `to = 2 * 365.25` â†’ `t_end = 0.0`. The same pattern appears at
line 1736 for intervention `at` times.

Fix: Use `Autodiff.simplify_fixpoint` before the match â€” it already does
constant folding. Or call `eval_const_expr` on the AST directly. Error if
the result is still non-constant instead of returning 0.0.

**#3 (High): ~25 `failwith` calls in parser.mly and expander.ml.**
Parser.mly (lines 67, 119, 156-189, 345-346, 434, 517, 595) and expander.ml
(lines 106-107, 1427, 1564-1690). Caught by `compiler.ml:63` so they don't
produce stack traces, but the error messages lack error codes, source
locations, and hints. Violates CLAUDE.md: "Never use `failwith` or
`assert false` for user-facing errors."

The time function failwiths are the worst â€” a user forgetting `step` in a
periodic forcing function gets "periodic with 'on' requires 'step' (bin
width)" with no source location. The parser ones are harder because the
parser doesn't have access to the diagnostics context.

**#4 (Medium): `ProjDerived(EIdent ...)` defaults to CumulativeFlow.**
Line 1831: bare identifier `projected = X` is assumed to be a transition
flow. If X is a compartment (user wants prevalence), they get silent wrong
behavior. Should check the name against `ctx.expanded_comp_tbl` / transition
names and either infer correctly or error if ambiguous.

**#5 (Low): Dead error variants in Rust validate.**
`WrongIndexCount` and `UnknownCompartmentInInitialConditions` are defined
but never emitted. Table lookup index count validation is never performed.

**#6 (Low): Keyword lookup is linear scan.**
`kw_table` is a 40-element association list searched linearly for every
identifier token. Should be a `Hashtbl`.

**#7 (Low): Lexer `pending_warnings` is a mutable global ref.**
Thread-unsafe, persists across compilations in the same process. Currently
drained in compiler.ml but fragile.

**#8 (Low): Coupling sugar removed but spec documents it.**
Language spec Â§10 documents `coupling[age = C_age]` extensively. The
implementation removed it. Spec should be updated to match.
