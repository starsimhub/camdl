## Engine Crate Review

### 1. Major DRY violation: `run_chain_binomial` vs `step_one`

`run_chain_binomial()` (the top-level simulation entry point) contains its own
inline Euler-multinomial loop that is a near-verbatim copy of `step_one()`. It
even re-declares `enum ResolvedDraw` locally (shadowing the module-level one).
The inline version lacks scratch-buffer reuse, lacks CPM z-value support, lacks
gamma tracking — it's a stale fork. This is the single biggest code smell in the
crate.

**Recommendation:** `run_chain_binomial` should call `step_one` in its inner
loop (with a `StepScratch`), exactly like all inference callers do. This would
delete ~80 lines and eliminate the divergence risk. The only thing the inline
version does differently is use `rng.gamma_multiplier()` without recording
gammas, which is fine — PGAS callers already handle that via
`scratch.gamma_used`.

### 2. Duplicated patch-population logic in `pgas.rs`

The "determine per-patch population for IVP compartments" block appears **three
times**: once in `complete_data_loglik`, once in `csmc_as` (as
`ivp_patch_pops`), and implicitly again via string-suffix matching. All three do
the same `rsplit('_').next()` + `ends_with(format!("_{}", suffix))` pattern.

**Recommendation:** Extract a
`fn patch_population(model, initial_counts, compartment_idx) -> i64` helper.
Call it from all three sites.

### 3. Duplicated resume-state z-value restoration

The name-based z-value reordering on resume (build `HashMap<&str, f64>` from
saved `param_names`, look up each current param) is copy-pasted identically
between `run_pgas` and `run_pmmh`. Both even have the same fallback warning
messages.

**Recommendation:** Extract to a shared helper like
`fn restore_transformed(saved_names: &[String], saved_z: &[f64], if2_params: &[EstimatedParam], current_params: &[f64]) -> Vec<f64>`.

### 4. Duplicated `eval_table_expr` vs `eval_expr`

`eval_table_expr` in `compiled_model.rs` is a subset of `eval_expr` from
`propensity.rs` — it handles `Const`, `Param`, `BinOp`, `UnOp` with identical
match arms. The BinOp dispatch is literally the same 16-arm match. The only
difference is `eval_table_expr` doesn't have access to compartment state (which
is correct — table values are param-only). But this could be handled by calling
`eval_expr` with a dummy zero-state `EvalCtx`, which you already construct for
parameterized ICs in `initial_state()`.

**Recommendation:** Delete `eval_table_expr`, use `eval_expr` with a zero-state
`EvalCtx` instead. Saves ~50 lines and eliminates the risk of the two BinOp
dispatches diverging.

### 5. `run_pgas` is 500+ lines — extract rung logic

`run_pgas` is a single function spanning ~550 lines. The per-rung NUTS path
alone is ~100 lines, the MH-within-Gibbs path another ~40, mass-matrix
adaptation ~60, replica exchange ~30, reporting ~40. This is hard to audit.

**Recommendation:** Extract at minimum:

- `fn nuts_theta_update(...)` — the NUTS θ|X step
- `fn mh_theta_update(...)` — the MH-within-Gibbs θ|X step
- `fn replica_exchange_sweep(...)` — the swap logic

The per-rung `Vec<Vec<...>>` state (8+ parallel vectors: `rung_params`,
`rung_transformed`, `rung_ll`, `rung_trajectory`, `rung_nuts_mass`,
`rung_nuts_step_size`, `rung_log_proposal_sd`, `rung_welford_*`) screams for a
`struct RungState`. Then you'd have `Vec<RungState>` instead of 12 parallel
vectors that you `.swap(i, j)` in sync.

### 6. `prior_log_density_and_grad_z` duplicates `Prior::log_density`

`pgas.rs` has `prior_log_density_and_grad_z` which re-implements the density
calculation that `Prior::log_density` (in `pmmh.rs`) already provides. The
gradient is new, but the density computation is duplicated.

**Recommendation:** Have `prior_log_density_and_grad_z` call
`Prior::log_density` for the value, then compute only the gradient. Or better:
add a `log_density_and_grad` method to `Prior` itself.

### 7. `obs_at_substep` HashMap rebuilt on every call

Both `complete_data_loglik` and `csmc_as` build a `HashMap<usize, usize>`
mapping substep→obs_idx from scratch each time they're called. For a 46K-substep
trajectory with 520 observations, this is a HashMap with 520 entries rebuilt on
every Gibbs sweep (thousands of times).

**Recommendation:** Precompute `obs_at_substep` once in `run_pgas` and pass it
as a parameter. It's invariant across sweeps (only depends on observation times
and dt).

### 8. Categorical sampling from log-weights: duplicated ad-hoc implementation

The "sample from categorical(softmax(log_weights))" pattern — compute max,
exponentiate, accumulate, sample uniform — appears at least 3 times in `csmc_as`
(ancestor sampling, final trajectory selection) and could easily be a utility
function. The degenerate-fallback logic also varies slightly between the two
sites.

**Recommendation:**
`fn sample_categorical_log(log_weights: &[f64], rng: &mut StatefulRng) -> Option<usize>`
returning `None` for all-`-inf`.

### 9. `ResolvedDraw` enum declared twice

There's a module-level `enum ResolvedDraw` in `chain_binomial.rs`, _and_ a local
one inside `run_chain_binomial`. The local one shadows the module-level one. The
module-level one is used by `StepScratch`. This is confusing.

**Recommendation:** Delete the local declaration in `run_chain_binomial` (which
should be calling `step_one` anyway per item #1).

### 10. Minor: `CompiledModel` has no `Debug` derive

`CompiledModel` is the god object of the system, but it doesn't implement
`Debug`, making debugging painful. `ResolvedModel` also lacks `Debug`. Consider
at least a manual `Debug` impl that prints field names/sizes without dumping all
data.

### 11. Minor: `transform_deriv` and `EstimatedParam` methods overlap

`transform_deriv` in `pgas.rs` computes dθ/dz, while
`EstimatedParam::log_jacobian` computes log|dθ/dz| and `jacobian_grad` computes
d/dz of that. These are all related but scattered. Consider consolidating
transform math into `EstimatedParam` methods: `deriv(z) -> f64`,
`log_jacobian(z) -> f64`, `log_jacobian_grad(z) -> f64`.

Overall the code quality is high — well-commented, correct invariants
documented, good debug assertions. The main theme is **convergent evolution**:
as PGAS, PMMH, and IF2 grew independently, shared patterns weren't extracted. A
refactoring pass focused on items #1–#5 would meaningfully reduce surface area
and divergence risk.
