## Compiler Crate Review

### 1. `days_per_unit` vs `days_per` — duplicated unit conversion with different values

`expander.ml` has **two** functions mapping time units to days:

- `days_per_unit` (line ~100): uses `365.2425` for Years/Months
- `days_per` (line ~340): uses `365.25` for Years, `30.4375` for Months

These are used in different places (`parse_date_to_float` uses `days_per_unit`;
`unit_to_model_time` uses `days_per`). The difference is small (365.2425 vs
365.25) but it means date-to-float and rate-conversion use slightly different
calendars. This is almost certainly a bug, not a design choice.

**Recommendation:** Pick one constant set and delete the other function.
`365.2425` is the Gregorian average; use it everywhere.

### 2. File I/O pattern duplicated 3 times

`load_table_data`, `read_dim_column_from_file`, and
`load_interpolated_for_level` all independently:

- resolve path via `resolve_data_path`
- check `Sys.file_exists`
- determine separator from extension (`.csv`→`,`, `.tsv`→`\t`)
- open file, read header, parse rows, close file
- emit E200/E205 on error

The separator-from-extension logic alone appears 3 times. The file-not-found +
open + read-header + row-loop + close pattern is repeated 3 times with minor
variations.

**Recommendation:** Extract a `read_csv_rows ctx path ~on_header ~on_row` helper
that handles path resolution, existence check, extension→separator, header
parsing, row iteration, and cleanup. Each caller provides just the
domain-specific `on_header`/`on_row` callbacks. This would save ~80 lines and
centralize error handling.

### 3. `serialize.ml` / `deserialize.ml` are pure boilerplate — consider ppx_deriving_yojson

These two files total ~950 lines and are entirely mechanical mappings between
OCaml types and JSON. Every time you add a field to `ir.mli`, you must update
both files in lockstep. This is a classic maintenance trap.

**Recommendation:** If you want to keep hand-written serde (for control over the
JSON shape, which matters since the Rust side reads it), at least co-locate the
`_to_json` and `_of_json` for each type in the same file so you can audit them
side-by-side. As-is, adding a field to e.g. `transition` requires edits in
`ir.mli`, `ir.ml`, `serialize.ml`, `deserialize.ml`, and the Rust
`transition.rs` — 5 files for one field.

Alternatively, `ppx_deriving_yojson` would auto-generate both directions,
cutting ~800 lines. The JSON shape would change slightly (ppx uses
`[@key "..."]` for field renaming), but since you control both sides of the
wire, that's manageable.

### 4. `expander.ml` context is a mutable bag — consider making passes explicit

The `context` record has 25 mutable fields and accumulates state across
`collect_declarations`, `resolve_dimensions`, `check_shadowing`, `check_guards`,
and `expand_*`. The flow dependencies between these passes are implicit. For
example, `expanded_comp_cache` is lazily populated by
`get_expanded_compartments`, which depends on `dim_registry` having been
populated by `resolve_dimensions`. If someone reorders calls in `expand_detail`,
things silently break.

**Recommendation:** This is a larger refactor, but even a minimal version —
making `expanded_comp_cache` non-optional (computed once in `expand_detail`
after `resolve_dimensions` and passed explicitly) — would make the dependency
clearer.

### 5. `resolve_expr` / `resolve_ident_name` — linear scans for every identifier

Every identifier lookup does `List.find_opt` or `List.exists` over
`ctx.let_bindings`, `ctx.comp_decls`, `ctx.param_decls`, `ctx.func_decls`, plus
`List.mem` over the full expanded compartment list. For a spatial model with
5000 compartments and 25,000 transitions, this is O(n²) in the expansion phase.

**Recommendation:** Build `Hashtbl` or `Set` lookups for `let_bindings`,
`param_decls`, `func_decls`, and expanded compartments once (after
`resolve_dimensions`), then use O(1) lookups in `resolve_expr`. This matters for
models like `polio_spatial_5` where expansion is already the slow phase.

### 6. `inspect.ml` reads source file twice

Both `run_inspect` and `run_check` independently open, read, and close the
source file with identical code blocks (8 lines each). They then call
`Compiler.compile_detail_result` which... reads it again internally (or uses the
string passed to it).

**Recommendation:** Factor out the file-read into a shared helper. Minor, but
the duplication is obvious.

### 7. `collect_declarations` reverses insertion order via `@` append

Every `DCompartments`, `DParameters`, etc. case does
`ctx.comp_decls <- ctx.comp_decls @ cs`. This is O(n²) in the number of
declarations — each append traverses the existing list. For small models it's
fine, but it's an unnecessary pattern.

**Recommendation:** Accumulate with `::` (prepend) and `List.rev` once at the
end of `collect_declarations`, or switch to a `Buffer`-like mutable accumulator.

### 8. `strides` array computed redundantly in `load_table_data`

The stride computation
`for i = n - 2 downto 0 do strides.(i) <- strides.(i+1) * ...` appears **twice**
in `load_table_data` — once for row parsing, once for the missing-entry
diagnostic. Same logic, same result.

**Recommendation:** Compute once, reuse.

### 9. Dead code in `inspect.ml`

`pp_indices` defines `pp_one` and then `pp_one_correct`, with an `ignore pp_one`
to suppress the unused warning. Just delete `pp_one`. Similarly,
`run_transition_count` has `ignore neq` at the bottom — `neq` is computed but
never used.

### 10. `autodiff.ml` simplifier doesn't iterate to fixpoint

The doc comment says "Applied to fixed point" but the code calls `simplify`
once. Nested simplification opportunities (e.g., `0 * (x + 0)` → after one pass
becomes `0 * x` → needs another pass to become `0`) are missed. The recursive
structure handles most cases since it recurses first then simplifies, but the
fixpoint claim is inaccurate.

**Recommendation:** Either add
`let rec simplify_fixpoint e = let e' = simplify e in if e' = e then e' else simplify_fixpoint e'`
or correct the comment. For PGAS gradient expressions, the trees are small
enough that this is a correctness/clarity issue, not a performance one.

## Deferred items

### #3: ppx_deriving_yojson (deferred)
Large migration: ppx generates different JSON field names by default.
Every field across ~40 types would need [@key "..."] annotations or
Rust-side serde renames. The co-location alternative (combining
serialize.ml and deserialize.ml per-type) is lower risk.

### #4: Explicit compiler passes (deferred)
The context record has 25 mutable fields with implicit ordering
dependencies. Full fix requires architectural change to expander.ml
(~2000 lines). Minimal version (non-optional expanded_comp_cache)
is doable but low priority.
