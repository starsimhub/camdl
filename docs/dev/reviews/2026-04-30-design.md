# Design quality review — 2026-04-30

Companion to `2026-04-30-correctness.md`. Covers Medium and Low severity
findings: type design, DRY, dead-guard code, and OCaml boundary issues that
don't immediately produce wrong numbers but carry extension and maintenance
risk.

Verified-against: HEAD `5627ef1`.

## Findings

### Medium

---

**M1. `HierarchicalPrior.kind` is a stringly-typed distribution name in both the
OCaml IR and the Rust IR.**

`ocaml/lib/ir/ir.ml:217–222`:

```ocaml
type hierarchical_prior = {
  hkind:      string;  (* "uniform", "normal", "log_normal", ... *)
  hargs:      (string * expr) list;
  hpool_over: string;
}
```

`rust/crates/ir/src/parameter.rs:56–60`:

```rust
pub struct HierarchicalPrior {
    pub kind:      String,  // "uniform", "normal", "log_normal", ...
    pub args:      std::collections::BTreeMap<String, crate::expr::Expr>,
    pub pool_over: String,
}
```

`PriorDist` (the non-hierarchical equivalent) is a proper enum with typed
variants in `ir/src/parameter.rs`. `HierarchicalPrior` uses a raw `String` for
the distribution kind and an `(string * expr) list` / `BTreeMap<String,
Expr>`
for arguments — no structural constraint that `kind = "beta"` implies `args` has
`alpha` and `beta`, not `mu` and `sigma`.

The Rust backend that consumes `hierarchical.kind` must match on the string at
runtime. If the string is unrecognised (e.g., `"log_normal"` vs `"lognormal"`),
the error fires at inference time, not at compile time.

**Why it matters.** A new distribution added to `PriorDist` (the flat case) has
to be added manually to the `HierarchicalPrior` handling code, with no compiler
assistance. The two prior representations are structurally inconsistent: the
flat case is type-safe; the hierarchical case is stringly-typed.

**Fix:** Define a `HierarchicalKind` enum (mirroring `PriorDist` variants) in
both the OCaml IR (`ir.ml`) and the Rust IR (`parameter.rs`). Use it for
`hkind`/`kind`. The `args` map can remain a `BTreeMap<String, Expr>` (the
argument names are inherently string-keyed in the expression language).

**M2. IR `schema.json` is missing three `Parameter` fields: `hierarchical`,
`param_kind`, `param_dim`.**

`ir/schema.json` defines the `Parameter` object but omits three fields that
exist in `ocaml/lib/ir/ir.ml` and `rust/crates/ir/src/parameter.rs`:

| Field          | OCaml type                  | Rust type                   | Schema |
| -------------- | --------------------------- | --------------------------- | ------ |
| `hierarchical` | `hierarchical_prior option` | `Option<HierarchicalPrior>` | absent |
| `param_kind`   | `string option`             | `Option<String>`            | absent |
| `param_dim`    | `(int * int) option`        | `Option<(i32, i32)>`        | absent |

CLAUDE.md §"Changing the IR schema" requires the schema to be the single source
of truth. Missing fields mean:

1. External tools that validate IR against the schema will reject valid IR
   (false positives).
2. The schema cannot be used to detect drift between the OCaml and Rust type
   definitions.
3. Hierarchical priors — a current production feature — have no schema-level
   contract.

**Fix:** Add the three fields to the `Parameter` definition in `schema.json`,
including their types and `"required"` / `"optional"` status. Run the schema
validation tests after update.

**M3. `if2.rs:523` panics when `n_iterations == 0`.**

```rust
// if2.rs:523
let last_iter = iterations.last().unwrap();
```

`iterations` is populated by the per-iteration loop at lines 271–520. If
`config.n_iterations == 0`, the loop body never executes, `iterations` is empty,
and `.last().unwrap()` panics. No validation prevents `n_iterations = 0` in
`IF2Config`.

`config.n_iterations` comes from `FitConfigV2`'s `[stages.scout]` TOML key,
which accepts any `usize`. A user who writes `n_iterations = 0` in an IF2 stage
gets a panic instead of a user-facing error.

**Fix:** Add to `IF2Config::validate()` (or `FitConfigV2::validate()`):

```rust
if config.n_iterations == 0 {
    return Err("IF2: n_iterations must be ≥ 1".into());
}
```

Then replace `.last().unwrap()` with
`.last().expect("n_iterations ≥ 1 enforced
at construction")` to document the
invariant.

**M4. `fit/runner.rs` `best_chain` index not bounds-checked before indexing into
results.**

`rust/crates/cli/src/fit/runner.rs` (line ~2190):

```rust
let best = &cr.results.iter().find(|(id, _)| *id == cr.best_chain).unwrap().1;
```

`cr.best_chain` is an index into `cr.results`. If the two get out of sync (e.g.,
if a chain's results are dropped due to an error mid-run and `best_chain`
retains the old index), `.find(...)` returns `None` and `.unwrap()` panics. The
panic fires after inference has completed, during the result-collection phase,
discarding all computed samples.

**Fix:** Replace with:

```rust
let best = cr.results.iter()
    .find(|(id, _)| *id == cr.best_chain)
    .ok_or_else(|| format!(
        "best_chain {} not found in results (chains: {:?})",
        cr.best_chain, cr.results.iter().map(|(id, _)| id).collect::<Vec<_>>()))?;
```

**M5. `simplex_groups` non-IF2 warning is `eprintln!`, not a proper
diagnostic.**

`rust/crates/cli/src/fit/config_v2.rs:1178–1180`:

```rust
eprintln!("\x1b[33mwarning:\x1b[0m fit declares simplex_groups, \
    but non-IF2 stage(s) {} do not currently honour the \
    constraint ...", names);
```

This warning fires during `validate()` and goes to stderr with ANSI escape
codes. Problems:

1. ANSI codes corrupt non-TTY output (CI logs, pipe to file).
2. The warning is not recorded in `run.json` or any audit trail; a user who runs
   with stdout redirected never sees it.
3. Severity ambiguity: ignoring simplex constraints in PGAS/PMMH is a
   parameter-space violation that affects posterior validity. A
   `stderr + eprintln!` is the wrong severity signal for something this
   consequential.

**Fix:** Use the `progress` module's warning API (which already handles
TTY-detection and structured output). Record the warning in `run.json`'s
`provenance` field. Consider making it a hard error until PGAS/PMMH simplex
support is implemented — users who declare simplex groups and run PGAS get
silently wrong results.

### Low

**L1. No `.mli` interface files for any OCaml library module.**

`ocaml/lib/ir/`, `ocaml/lib/compiler/`, `ocaml/lib/expand/` contain 14+ `.ml`
files with zero corresponding `.mli` files. All internal types and functions are
accessible to any module in the build. The OCaml module system's principal value
— constraining callers to a declared interface — is unused throughout.

Consequences:

- No structural enforcement that callers use `Ir.parameter` through its
  documented API. Any module can read `p.hierarchical` without going through the
  invariant-enforcing accessor path.
- Functions meant to be internal (`prior_classification`, helper extractors in
  `expander.ml`) are callable from anywhere in the build.
- Future changes to IR types get no "who touched this" guidance from the
  compiler.

**Fix:** Add `.mli` files for at minimum `ir/ir.mli` and `ir/serde.mli` (the
public API of the IR layer). For the compiler layer, `compiler.mli` exposing
only `compile : Ast.t -> (Ir.t, Diagnostic.t list) result`.

---

**L2. `fit/config_v2.rs:687` uses `unwrap_or(json!({}))` on JSON serialization
of a live struct.**

```rust
serde_json::to_value(self).unwrap_or(json!({}))
```

`serde_json::to_value` fails only if the struct contains non-serializable values
(NaN/Inf `f64`, non-string map keys). `FitConfigV2` holds `f64` fields (bounds,
rw_sd) that can in principle be NaN — e.g., if a TOML parser produced one. In
that case the serialization silently returns `{}`, and whatever consumes the
`json!({})` (likely a hash computation or provenance record) produces a garbage
value with no diagnostic.

**Fix:** Propagate the error:

```rust
serde_json::to_value(self).map_err(|e| format!("config serialization failed: {}", e))?
```

**L3. Rand-distr `Gamma::new()` and `Exp::new()` in prior sampling use
`.unwrap()` on parameters that could theoretically be invalid.**

`rust/crates/cli/src/main.rs` (around lines 1300–1319):

```rust
let g = rand_distr::Gamma::new(p.alpha, 1.0).unwrap();
let g = rand_distr::Gamma::new(p.shape, 1.0 / p.rate).unwrap();
let e = rand_distr::Exp::new(p.rate).unwrap();
```

IR validation should prevent `alpha ≤ 0`, `shape ≤ 0`, or `rate ≤ 0` from
reaching these call sites. But if an IR arrives with invalid parameter values
(hand-crafted, or from a future compiler bug), the panic fires inside the prior
sampling loop with no user-facing message and no indication of which parameter
caused the failure.

**Fix:** Replace each `.unwrap()` with `.map_err(|e| SimError::...)` and
propagate, or add a pre-flight check:
`ensure!(p.alpha > 0.0,
"Gamma prior alpha must be positive; got {}", p.alpha)`.

**L4. `validate_no_duplicates` is the only caller that checks for the
malformed-range case, and it does so after calling `to_vec()`.**

This is the lower-severity aspect of H2 (see correctness review). Even with H2
fixed, `validate_no_duplicates()` should document that it is the canonical
validation entry point for `SeedsSpec` and should be called before any call to
`to_vec()`. The current structure allows callers to call `to_vec()` in isolation
(e.g., in tests or CLI preprocessing) and silently receive empty vectors.

**L5. Resume-to-extend is not ergonomic: editing `sweeps` in `fit.toml` creates
a new fit directory, making the existing `resume_state.bin` invisible.**

The fit directory is keyed on raw `fit.toml` bytes (`fit_content_hash`), while
the resume-state file is keyed on the stage's `identity_payload` (algorithmic
knobs only). If a user wants to extend a run from 500 → 1000 sweeps, they must
edit `fit.toml`, which changes `fit_content_hash`, which creates a new fit
directory, and the original `resume_state.bin` never appears in the new path.

This was documented in `b7b50f53` as a known caveat: "editing `sweeps` and
re-running creates a new fit_dir — the resume_state.bin in the original dir
isn't visible." The identity-vs-extension split is at the stage-hash level, not
the fit-content-hash level.

**Fix:** A `--sweeps N` CLI flag (analogous to `--max-tree-depth N`) that
overrides the TOML value without changing the on-disk TOML, so
`fit_content_hash` stays stable while the extension request is honoured. This is
a feature request, not a correctness issue.

## Positive findings

**Output tree design is clean.** `run_paths::output_root()` implements a correct
four-layer precedence (CLI > config > `CAMDL_OUTPUT_DIR` > default) with tests
for all four cases and a `Mutex`-serialized env-var test to prevent
parallel-test races. Hash truncation uses `&hash[..8.min(hash.len())]` — no
panic on short hashes.

**Atomic `run.json` writes prevent split-brain on crash.** `Run::write()` in
`run_meta.rs` writes to `run.json.tmp` then renames. If the process dies
mid-write, readers see `CacheStatus::Miss` rather than a partial file. Test at
lines 635–649 verifies no `.tmp` residue after success; test at lines 653–675
simulates the crash window.

**PGAS and PMMH algorithmic knobs are now fully wired.** The 12 knobs surfaced
in `a5614846` (`tempering`, `max_tree_depth`, `csmc_sweeps_per_nuts`,
`n_trajectories`, `dense_mass`, `use_nuts` for PGAS; `adapt`, `adapt_start`,
`rho`, `n_trajectories` for PMMH) are correctly threaded from `FitConfigV2`
through `PgasStageOpts`/`PmmhStageOpts` to `PGASConfig`/`PMMHConfig`. Each has a
documented `default_*()` function. None silently override user-supplied values.

**`RunStatus` typed enum eliminates sentinel.** The previous
`wall_time_seconds == 0.0` pattern is gone. `is_running()` /
`wall_time_seconds()` return `bool` / `Option<f64>` respectively. Serialization
round-trip is tested.

**Log-space discipline is correct throughout the inference stack.**
`log_sum_exp` handles all edge cases (all-`−∞`, `+∞`). ESS is computed in log
space. Systematic resampling takes a single uniform draw. `LOG_PROB_FLOOR` is a
named constant with a comment explaining the choice. No catastrophic
cancellation sites found.

**Prior-types consolidation is complete.** `sampling::PriorSpec` is deleted; all
prior handling goes through `ir::parameter::PriorDist`. `prior_spec_to_prior` is
deleted; `resolve_prior` calls `Prior::from_ir` directly. The
`config_v2::PriorDist` re-export preserves downstream `use` paths.
`CLEANUP-prior-types.md` and the `TYPES-REFERENCE.md` are updated.
