# PMMH Implementation Plan for camdl

## Overview

Add Particle Marginal Metropolis-Hastings (PMMH) to camdl's inference toolkit.
PMMH uses an outer Metropolis-Hastings loop with the bootstrap particle filter as
an unbiased likelihood estimator, enabling full posterior exploration of multimodal
surfaces that IF2's cooling schedule cannot traverse.

**Estimated scope:** ~1000 lines of new Rust + ~300 lines of tests.
Most infrastructure (PF, dmeasure, transforms, trace IO) already exists.

## Phase 1: Core PMMH Engine (`sim/src/inference/pmmh.rs`)

### Algorithm

```
Input: θ_init, Σ_proposal, n_steps, n_particles, prior
θ ← θ_init  (on transformed scale)
ll ← pfilter(from_transformed(θ), n_particles)

for step in 1..n_steps:
    θ' ← θ + N(0, Σ_proposal)           # propose on transformed scale
    θ'_natural ← from_transformed(θ')    # back to natural for PF
    if any(θ'_natural out of bounds):
        reject (ll' = -∞)
    else:
        ll' ← pfilter(θ'_natural, n_particles)
        log_prior' ← log_prior(θ'_natural)
        log_prior  ← log_prior(θ_natural)
        log_α ← (ll' + log_prior') - (ll + log_prior)
                 + log_jacobian(θ') - log_jacobian(θ)   # transform correction
        if log(U) < log_α:
            θ ← θ', ll ← ll'                            # accept

    record (step, θ_natural, ll, accepted)
```

### Key design decisions

**1. Proposals on transformed scale, Jacobian correction included.**
Reuse `IF2Param::to_transformed` / `from_transformed`. The Jacobian correction
for log-transform is `log(θ)` (cancels in the ratio for symmetric RW proposals
on transformed scale — but we include it for correctness with logit).

**2. PF seed varies per step.**
Each MH step uses a different PF seed: `base_seed + step`. This prevents the
"correlated pseudo-marginal" artifact where the same PF randomness inflates
acceptance. The PF is already deterministic given a seed, so this is clean.

**3. No cooling, no parameter perturbation inside the PF.**
The PF runs at a single fixed θ per step. This is the fundamental architectural
difference from IF2.

### Types and API

```rust
// sim/src/inference/pmmh.rs

/// Prior distribution for one parameter.
#[derive(Clone, Debug)]
pub enum Prior {
    /// Flat (improper) prior on [lo, hi]. Log-density = 0 within bounds.
    Flat,
    /// Normal(mean, sd) on the NATURAL scale.
    Normal { mean: f64, sd: f64 },
    /// Normal(mean, sd) on the TRANSFORMED scale (log or logit).
    TransformedNormal { mean: f64, sd: f64 },
}

/// PMMH configuration.
pub struct PMMHConfig {
    pub n_steps: usize,
    pub n_particles: usize,
    pub dt: f64,
    /// Initial proposal covariance on the transformed scale.
    /// Diagonal: proposal_sd[i]² for each estimated parameter.
    pub proposal_sd: Vec<f64>,
    /// Enable adaptive Metropolis (Haario et al. 2001).
    /// After `adapt_start` steps, Σ is updated from chain history.
    pub adapt: bool,
    pub adapt_start: usize,
    /// Thin: only record every `thin`-th step.
    pub thin: usize,
    /// Burn-in: discard first `burn_in` steps from output.
    pub burn_in: usize,
}

/// One recorded MCMC step.
#[derive(Clone, Debug)]
pub struct PMMHStep {
    pub step: usize,
    pub params: Vec<f64>,       // natural scale, full param vector
    pub log_likelihood: f64,    // PF estimate
    pub log_prior: f64,
    pub accepted: bool,
}

/// Result of a PMMH run.
pub struct PMMHResult {
    pub steps: Vec<PMMHStep>,
    pub acceptance_rate: f64,
    pub n_steps: usize,
    /// Best (highest posterior) parameter set found.
    pub map_params: Vec<f64>,
    pub map_loglik: f64,
}
```

### Core function signature

```rust
/// Run PMMH. The `eval_loglik` closure runs a PF and returns log L̂(θ).
/// This keeps the engine decoupled from CompiledModel / chain_binomial.
pub fn run_pmmh(
    if2_params: &[IF2Param],
    priors: &[Prior],
    base_params: &[f64],        // full param vector (fixed + estimated)
    config: &PMMHConfig,
    eval_loglik: &dyn Fn(&[f64], u64) -> f64,  // (params, seed) → log L̂
    seed: u64,
    on_step: Option<&dyn Fn(usize, f64, bool)>,  // (step, loglik, accepted)
) -> PMMHResult
```

The `eval_loglik` closure is built in the CLI layer from `run_quick_pfilter`.
This separation matches how IF2's `step_fn` / `project_fn` / `dmeasure_fn`
closures are built in the CLI and passed to the engine.

### Adaptive Metropolis (Haario et al. 2001)

After `adapt_start` accepted steps, update the proposal covariance:

```
Σ_proposal = (2.38² / d) × Cov(θ_{1:n}) + ε × I_d
```

where `d` is the number of estimated parameters and `ε = 1e-6` prevents
degeneracy. This is the standard "AM" algorithm. The `2.38² / d` scaling
is optimal for Gaussian targets (Roberts & Rosenthal 2001).

Implementation: maintain running mean and sum-of-squares matrix, update
incrementally (Welford's algorithm). No matrix decomposition needed since
we only need to sample from N(0, Σ) — use Cholesky of Σ, updated every
100 steps (not every step, to amortize the O(d³) cost; d ≤ 15 here so
it's negligible anyway).

### Orchestration in `sample.rs`

```rust
pub fn run_sample(fit: &FitToml, starts_from: Option<&str>, seed: u64, force: bool) -> Result<(), String> {
    let stage_dir = format!("{}/sample", fit.fit.output_dir);
    let sc = fit.sample.as_ref();

    // 1. Load prior fit_state if --starts-from provided
    let prior_state = starts_from.map(|dir| FitState::load(dir)).transpose()?;

    // 2. Build FitRunConfig (reuse existing builder)
    let n_chains = sc.and_then(|s| s.chains).unwrap_or(4);
    let n_steps = sc.and_then(|s| s.steps).unwrap_or(50_000);
    let n_particles = sc.and_then(|s| s.particles).unwrap_or(2000);

    let config = FitRunConfig::build(
        fit, prior_state.as_ref(),
        n_chains, n_particles, 1,  // iterations=1, unused for PMMH
        1.0,  // cooling unused
        seed, false,
    )?;

    // 3. Build proposal covariance
    //    Priority: scout empirical covariance > rw_sd from [estimate]
    let proposal_sd = build_proposal_sd(&config, sc, starts_from)?;

    // 4. Build priors (flat by default; future: [prior] section in fit.toml)
    let priors: Vec<Prior> = config.if2_params.iter()
        .map(|_| Prior::Flat)
        .collect();

    // 5. Build the loglik evaluator closure
    let eval_loglik = |params: &[f64], pf_seed: u64| -> f64 {
        run_quick_pfilter(&config, params, n_particles, pf_seed)
    };

    // 6. Run chains in parallel (same pattern as run_chains)
    let results: Vec<(usize, PMMHResult)> = (0..n_chains)
        .into_par_iter()
        .map(|chain_id| {
            let chain_seed = seed ^ (chain_id as u64).wrapping_mul(0x9e3779b97f4a7c15);
            let result = run_pmmh(
                &config.if2_params, &priors, &config.base_params,
                &pmmh_config, &eval_loglik, chain_seed, Some(&progress_cb),
            );
            (chain_id, result)
        })
        .collect();

    // 7. Diagnostics: Gelman-Rubin R̂, acceptance rates
    // 8. Write traces, summary, fit_state
    // 9. Report
}
```

### Proposal covariance seeding

When `proposal_from` points to a scout directory, read all chain traces
and compute the empirical covariance of the final-iteration parameters
across chains. Scale by `2.38² / d`. This gives a much better starting
proposal than the rw_sd values (which are tuned for IF2 perturbations,
not MH jumps).

```rust
fn build_proposal_sd(
    config: &FitRunConfig,
    sc: Option<&SampleConfig>,
    starts_from: Option<&str>,
) -> Result<Vec<f64>, String> {
    // Try to load scout chain endpoints for empirical covariance
    if let Some(dir) = sc.and_then(|s| s.proposal_from.as_deref()).or(starts_from) {
        if let Ok(cov) = load_scout_covariance(dir, &config.if2_params) {
            let d = config.if2_params.len() as f64;
            let scale = 2.38_f64.powi(2) / d;
            return Ok(cov.iter().map(|&v| (v * scale).sqrt()).collect());
        }
    }
    // Fallback: use rw_sd from [estimate], scaled up
    // IF2 rw_sd is per-step; PMMH needs per-proposal (larger)
    Ok(config.if2_params.iter().map(|p| {
        p.transformed_sd(p.rw_sd, p.initial) * 5.0  // heuristic: 5× IF2 step size
    }).collect())
}
```

## Phase 3: Diagnostics

### PF variance calibration (preflight)

Before starting PMMH, run the PF 20 times at the starting θ and report
the SD of log L̂. This is critical for tuning `n_particles`:

```
pfilter variance check (20 replicates at starting params):
  log L̂ mean = -1844.2, sd = 1.8
  target sd: 1-3 (good), >5 (increase particles), <0.5 (can reduce particles)
```

If sd > 5, automatically suggest doubling particles. If sd < 0.5, suggest
halving (saves 2× compute). Add `--check-variance` flag to skip the
actual PMMH run and only do this calibration.

### Gelman-Rubin R̂

Compute R̂ across chains on the post-burn-in samples. Same algorithm as
`compute_rhat` in `runner.rs` but adapted for MCMC samples (not IF2
iteration means). Report per-parameter R̂ with the standard 1.1 threshold.

### Effective sample size (ESS)

Compute ESS from the autocorrelation of the chain. Use the initial
monotone sequence estimator (Geyer 1992) which is standard for MCMC:

```rust
fn mcmc_ess(chain: &[f64]) -> f64 {
    let n = chain.len();
    let mean = chain.iter().sum::<f64>() / n as f64;
    let var = chain.iter().map(|&x| (x - mean).powi(2)).sum::<f64>() / n as f64;
    if var < 1e-30 { return 1.0; }

    let mut sum_rho = 0.0;
    for lag in 1..n {
        let rho: f64 = chain.iter().zip(chain.iter().skip(lag))
            .map(|(&a, &b)| (a - mean) * (b - mean))
            .sum::<f64>() / (n as f64 * var);
        if lag >= 2 && rho < 0.0 { break; }  // initial positive sequence
        sum_rho += rho;
    }
    n as f64 / (1.0 + 2.0 * sum_rho)
}
```

### Acceptance rate monitoring

Track and report the running acceptance rate. Optimal for RWM on a
d-dimensional target is ~0.234 (Roberts et al. 1997). If acceptance
rate is:
- < 0.10: proposal too wide, suggest reducing `proposal_sd` or increasing particles
- > 0.50: proposal too narrow, suggest increasing `proposal_sd`
- 0.15-0.35: good range

With adaptive Metropolis, this should auto-correct after the adaptation
period.

## Risk Assessment

| Risk | Mitigation |
|---|---|
| PF variance too high for He model → sticky chains | Preflight variance check; auto-suggest particle count |
| Multimodal posterior → chains don't mix between modes | Expected! This is diagnostic information. Report per-chain MAP and flag mode disagreement. |
| Adaptive proposal doesn't converge within budget | Large adapt_start default (1000); fallback to fixed proposal with warning |
| Transform Jacobian errors | Log and logit Jacobians are simple; unit test each. The symmetric RW on transformed scale means Jacobians cancel in the ratio for log, but not for logit — must include for correctness. |
| Slow: 50k PF runs × 2000 particles | ~1.5 hours for window data on single core. 4 parallel chains = same wall time. Full data ~5 hours — overnight run. Acceptable. |

## Future Extensions (not in this plan)

- **Correlated pseudo-marginal MCMC** (Deligiannidis et al. 2018): correlate PF randomness across MH steps for better mixing. High ROI but complex implementation.
- **Parallel tempering**: run multiple chains at different "temperatures" (inflated observation noise) and swap — helps cross between modes.
- **Prior specification in fit.toml**: `[prior]` section with per-parameter distributions. Currently all flat.
- **PGAS (Particle Gibbs with Ancestor Sampling)**: alternative to PMMH with better mixing for the latent states. Requires TransitionDensity trait (placeholder already exists in mod.rs).
- **Profile likelihoods via PMMH**: use PMMH samples to construct profiles by marginalizing, rather than optimizing. More principled than IF2-based profiles.
